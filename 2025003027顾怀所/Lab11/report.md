# Lab11 实验报告：为 Sports 应用添加大屏自适应布局

## 1. WindowSizeClass 概念简介
`WindowSizeClass` 是 Jetpack Compose Material3 提供的窗口尺寸分类工具，用于根据设备屏幕尺寸动态调整布局。其中 `WindowWidthSizeClass` 将宽度分为三类：
- **Compact（紧凑）**：宽度 < 600dp，典型手机竖屏
- **Medium（中等）**：600dp ≤ 宽度 < 840dp，典型手机横屏/小平板
- **Expanded（展开）**：宽度 ≥ 840dp，典型平板/大屏设备

本实验中主要使用 `Expanded` 判断大屏设备，采用双窗格布局；其余情况使用单窗格布局。

## 2. SportsContentType 枚举设计思路
`SportsContentType` 定义了两种内容展示类型：
- `ListOnly`：仅显示列表（小屏模式），用户需在列表和详情页之间切换
- `ListAndDetail`：同时显示列表和详情（大屏模式），充分利用屏幕空间

设计理由：
- 枚举化内容类型可清晰区分不同布局模式，降低逻辑耦合
- 便于在 `SportsApp` 中通过 `when` 表达式快速切换布局逻辑
- 符合单一职责原则，每种类型对应明确的布局行为

## 3. SportsListAndDetails 布局设计说明
### 核心布局结构
使用 `Row` 作为容器实现并排布局，通过 `Modifier.weight()` 分配宽度比例：
- 左侧列表：`weight(1f)`，占总宽度 1/3
- 右侧详情：`weight(2f)`，占总宽度 2/3

### 比例分配理由
- 列表仅需展示条目概览，无需过多宽度，1/3 足以清晰显示所有列表项
- 详情页包含横幅图、标题、详细描述等内容，需要更多空间（2/3）保证阅读体验
- 1:2 的比例符合 Material Design 列表-详情布局规范，兼顾视觉平衡和内容展示优先级

### 交互逻辑
- 列表项点击时，仅更新右侧详情页的 `currentSport` 状态，无需页面跳转
- 大屏模式下始终显示列表和详情，用户操作更直观，无需返回列表页

## 4. SportsAppBar 大屏/小屏行为差异设计考虑
### 标题逻辑
- 大屏模式（ListAndDetail）：始终显示 "Sports"（列表标题），因为用户始终能看到列表，无需切换标题
- 小屏模式（ListOnly）：列表页显示 "Sports"，详情页显示 "Sport Info"，符合用户当前页面认知

### 返回按钮逻辑
- 大屏模式：隐藏返回按钮，因为用户已在主界面（列表+详情），返回操作无意义
- 小屏模式：详情页显示返回按钮（允许返回列表），列表页隐藏返回按钮

设计依据：
- 符合用户心智模型：大屏下无页面跳转，无需返回；小屏下有页面切换，需要返回操作
- 遵循 Material3 顶部应用栏设计规范，减少不必要的交互元素

## 5. 返回键处理策略
### 小屏模式
- 详情页：按返回键回到列表页（通过 `BackHandler` 调用 `navigateToListPage()`）
- 列表页：按返回键退出应用（默认系统行为）

### 大屏模式
- 无论点击哪个区域，按返回键直接退出应用（通过 `(context as Activity).finish()`）

差异原因：
- 小屏模式存在页面切换（列表 ↔ 详情），返回键需遵循导航层级
- 大屏模式无页面切换，列表和详情始终可见，返回键应执行退出应用的核心操作
- 符合 Android 系统导航规范，避免用户在大屏模式下无意义的返回操作

## 6. 实验中遇到的问题与解决过程
### 问题1：大屏预览不显示内容
- 现象：`SportsListAndDetailsPreview` 预览空白
- 原因：`SportsTheme` 未实现，仅保留了 TODO 注释
- 解决：补充 `SportsTheme` 实现，包裹 `MaterialTheme` 保证预览正常渲染

### 问题2：大屏模式下详情页返回按钮仍可点击
- 现象：大屏模式下详情页的 `BackHandler` 仍会响应
- 原因：`SportsDetail` 组件内部的 `BackHandler` 未区分布局模式
- 解决：大屏模式下给 `onBackPressed` 传空实现，同时在 `SportsListAndDetails` 中单独处理返回键逻辑

### 问题3：窗口尺寸计算后布局未实时更新
- 现象：折叠屏切换尺寸后布局未自动调整
- 原因：初始代码未监听窗口尺寸变化
- 解决：`calculateWindowSizeClass` 会自动感知窗口变化，Compose 重组机制会重新计算 `contentType`，无需额外处理

### 问题4：大屏模式下内边距不一致
- 现象：大屏模式下列表和详情的内边距混乱
- 原因：未统一处理 `contentPadding` 和自定义内边距
- 解决：在 `SportsListAndDetails` 的 `Row` 上统一设置内边距，保证布局一致性

## 7. 实验总结
本次实验成功实现了 Sports 应用的大屏自适应布局，核心收获：
1. 掌握了 `WindowSizeClass` 的使用方法，能根据屏幕尺寸动态调整布局
2. 理解了列表-详情（List-Detail）布局模式的设计原则和实现方式
3. 学会了 Compose 中处理不同布局模式下的导航和交互逻辑
4. 加深了对 Compose 重组、状态管理和 BackHandler 的理解

自适应布局提升了大屏设备的用户体验，符合现代 Android 应用设计规范，也为后续适配折叠屏、多窗口模式打下了基础。
